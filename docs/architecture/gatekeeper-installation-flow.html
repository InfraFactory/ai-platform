01
flux bootstrap
github.com/InfraFactory/ai-platform · path: infrastructure/clusters/local
GitRepository flux-system Kustomization SSH deploy key
what this does Installs the four Flux controllers into the cluster and commits gotk-components.yaml + gotk-sync.yaml into the repo. From this point Flux is self-managing — it watches its own GitRepository and will restore any deleted controller from Git.
⚠ flux-system/ in the repo is Flux-owned. Do not edit manually.
02
cluster kustomization reconciled
infrastructure/clusters/local/kustomization.yaml
opa-gatekeeper.yaml policies.yaml
what this does The cluster-level kustomization is the entry point Flux reads first. It references two Flux Kustomization CRs — one for the Gatekeeper platform resources, one for the policy chain. These are not kustomize files; they are Flux reconciliation objects.
03
opa-gatekeeper resources applied
infrastructure/platform/opa-gatekeeper/
Namespace HelmRepository HelmRelease Config CR
critical: targetNamespace The HelmRelease must declare targetNamespace: gatekeeper-system explicitly. Omitting it deploys all chart resources into flux-system, where Gatekeeper's webhook then intercepts Flux's own reconciliation operations.
⚑ N-1
04
chart fetched by helm-controller
open-policy-agent.github.io/gatekeeper/charts
gatekeeper@3.22.2 install.crds: Create upgrade.crds: CreateReplace
version pinning Chart version is pinned to 3.22.2 in the HelmRelease spec. crds: CreateReplace is required on upgrade — Gatekeeper ships CRDs inside the chart rather than separately, so Helm must be told to update them on each chart upgrade, not just on first install.
05
Helm installs into gatekeeper-system
pods scheduled · CRDs registered
gatekeeper-controller-manager gatekeeper-audit ConstraintTemplate CRD
two distinct processes controller-manager handles real-time admission — it intercepts API server requests and enforces deny/warn decisions synchronously. audit runs on a configurable interval (60s here), scanning existing resources for policy violations and populating .status.violations on each Constraint object.
06
admission webhooks registered
API server begins routing admission requests to Gatekeeper
ValidatingWebhookConfiguration MutatingWebhookConfiguration check-ignore-label webhook
webhook failure modes Gatekeeper registers with failurePolicy: Ignore by default in dev mode — a crashed Gatekeeper pod does not block all cluster operations. In production, failurePolicy: Fail is recommended for security-critical policies. The check-ignore-label webhook fires on every namespace operation to check for exemption labels.
07
namespace exemptions active
config.gatekeeper.sh/v1alpha1 · name: config
flux-system gatekeeper-system kube-system monitoring
why exemptions are not optional The Gatekeeper Config CR excludes namespaces from webhook interception entirely. Without flux-system in the exclusion list, Gatekeeper's webhook fires against Flux's own dry-run operations during reconciliation — a misbehaving webhook can make GitOps-based recovery impossible.
⚑ N-2
08
ConstraintTemplates kustomization synced
policies/templates/ · dependsOn: opa-gatekeeper
RequireResourceLimits DenyPublicLoadBalancer RestrictModelNamespace
dependsOn is mandatory The platform-policies-templates Kustomization declares dependsOn: opa-gatekeeper. Without this, Flux may attempt to apply ConstraintTemplates before the ConstraintTemplate CRD itself exists, producing an unresolvable API error. Each template compiles its Rego bundle into the OPA engine — the CRD must pre-exist.
09
Constraints kustomization synced
policies/constraints/ · dependsOn: platform-policies-templates
warn: require-resource-limits deny: deny-public-loadbalancer deny: restrict-model-namespace
enforcement action rationale warn on resource limits — existing platform pods (kube-prometheus-stack) may not declare limits; a hard deny would block the observability stack on day one. Switch to deny once all platform workloads are compliant. deny on LoadBalancer and model namespace — both are targeted enough that a false positive requires a deliberate misconfiguration.
✓ V-1
Operational Notes Recorded during deployment · InfraFactory/ai-platform · 2026-04-30
⚑ N-1
targetNamespace omission causes webhook deadlock
Omitting targetNamespace from the HelmRelease deploys Gatekeeper into flux-system. The admission webhook registers against the wrong service endpoint and begins intercepting Flux's own dry-run operations, returning 502 errors that block the reconciliation needed to fix the misconfiguration — a self-reinforcing deadlock recoverable only by manually deleting both webhook configurations.
→ docs/rca/rca-gatekeeper-flux-bootstrap.md
⚑ N-2
Missing flux-system exemption reproduces the same deadlock
The Gatekeeper Config CR's excludedNamespaces list must include flux-system. Without it, the check-ignore-label webhook fires against Flux's namespace dry-runs. A misbehaving Gatekeeper webhook in a GitOps environment can make the GitOps controller itself unable to remediate — the tool designed to self-heal loses the ability to do so.
→ docs/rca/rca-gatekeeper-flux-bootstrap.md
✓ V-1
All three constraints validated against live cluster
require-resource-limits — pod admitted with warnings, violations recorded in .status.violations on the Constraint object.
deny-public-loadbalancer — Service of type LoadBalancer denied in default namespace; error message includes permitted namespace list.
restrict-model-namespace — Pod with component=model label denied in default; admitted in ai-workloads.
